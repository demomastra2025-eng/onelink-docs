## Getting Started Paths

Choose your path based on what you want to accomplish:

### I want to understand the OneLink direction
Start with the [Domain and Access Architecture](/contributing-guide/domain-access-architecture) document. It captures:
- the account-scoped access model inherited from the upstream base platform
- the target platform layering for OneLink
- the planned split between shared platform logic and domain zones like healthcare and construction
- the CRM direction for shared entities, custom attributes, and domain-specific behavior

### I want to work on the product
Read the [Development Guide](/contributing-guide) to:
- Set up your development environment
- Follow the `onelink-main` branch workflow
- Understand upstream sync expectations
- Apply platform vs domain architecture rules while shipping features

Then use:

- [Project Operations](/contributing-guide/project-operations) for whole-project runtime and delivery flow
- [AI Agent Operating Model](/contributing-guide/ai-agent-operating-model) for correct repo and commit boundaries
- [Skill Map](/contributing-guide/skill-map) to understand which project skill should own which task

### I want to deploy or configure the product
Use the [Installation & Setup](/self-hosted) section. Much of this content is inherited from the upstream deployment base and remains operationally relevant for OneLink:
- Docker
- Kubernetes
- cloud providers
- environment configuration
- monitoring and maintenance

### I want to build integrations or extend APIs
Use the [API Reference](/api-reference/introduction) and the development docs for:
- application APIs
- platform APIs
- client APIs
- integration-facing implementation patterns

## What OneLink Is

OneLink is a product fork built on top of an existing omnichannel support platform. The goal is not only to rebrand the upstream product, but to evolve it into a shared platform that can support:

- shared support and CRM capabilities
- white-label branding and product-specific workflows
- isolated domain zones such as healthcare and construction
- controlled divergence from upstream while preserving the ability to receive upstream updates

## Documentation Principles

This docs folder should increasingly reflect OneLink itself:

- keep inherited operational docs where they still apply
- add OneLink-specific architecture and product direction first
- document where the fork intentionally diverges from upstream
- prefer OneLink terminology and workflow over generic upstream wording for active development docs
