# Internal AI Stack Native Runtime Plan

## Goal

This document defines the target AI stack for `onelink` and maps which parts of the local RubyLLM ecosystem should be adopted directly, adapted into product-native layers, or explicitly ignored for now.

The target is not "support every feature that exists in RubyLLM".

The target is:

- one logical AI runtime for assistant, copilot, tasks, onboarding, embeddings, transcription, and moderation
- one native contract for prompts, messages, tools, structured output, traces, and safety
- one observability model that supports debugging, operations, cost control, and evals
- a pragmatic scope that closes real `onelink` use cases without importing unnecessary library breadth

## Scope

This plan covers:

- core `ruby_llm` runtime usage
- `ruby_llm-schema` usage
- built-in and custom tools
- future MCP integration boundaries
- tracing and observability
- monitoring and alerting
- base and custom moderation
- evals and regression testing

This plan does not aim to:

- port `RubyLLM::Agent` directly into `onelink`
- mount third-party Rails engines for AI dashboards
- add image generation or wide provider breadth unless product needs it
- reach full parity with every neighboring repository in `/Users/akhanbakhitov/Documents/zeroprompt/ruby`

## Local Verification Assumptions

For local development and verification in `onelink`, the baseline infra assumption should stay explicit:

- Postgres runs in Docker
- Redis runs in Docker
- the Rails app itself runs directly on the developer machine, not inside Docker, in the standard local dev loop
- AI runtime changes should be verified against the same Docker-backed local services used by the app test and dev environments

This matters because AI-stack work often looks like pure Ruby code but still depends on Rails boot, Sidekiq-adjacent runtime wiring, cached state, and database-backed Captain entities.

## Progress Status

The following slices are already implemented in the current `onelink` codebase iteration:

- capability-aware runtime gating for thinking, structured output, tool calling, and multimodal input
- stricter structured-output lifecycle with schema validation, repair retry, and controlled fallbacks
- a shared `Llm::MessageFormat` layer used by shared `Llm` flows and Captain runtime for history restore and multimodal serialization
- explicit moderation failure-mode handling with `fail_open` and `fail_closed` semantics instead of silent skip behavior
- an initial `Llm::EventBus` foundation for normalized moderation events and future AI observability wiring
- Captain runtime event-bus callbacks for run, chat, tool, and handoff events
- schema invalid/repair event publication and safety-blocked events for moderation failures
- persisted `llm_events` ingestion with subscriber installation, derived cost estimation, and a basic metrics snapshot service
- an admin-only Captain observability read-path that exposes account-scoped AI snapshots and filtered recent `llm_events`
- a first `Llm::SafetyPolicy` layer that wraps provider moderation and adds deterministic account-configurable safety blocklists
- tool-level safety enforcement for assistant runtime tools, copilot tools, and custom-tool preview execution on `tool_arguments` and `tool_results`
- a product-native release gate over persisted `llm_events`, exposed through the observability API and configurable through `captain_runtime.release_gate`
- sanitized `llm_events` payload persistence so observability data is truncated and secret-bearing fields are redacted before storage
- autonomous agent high-risk tool gating through `captain_runtime.agent_high_risk_tools` and `agent_high_risk_tool_ids`
- Prometheus-compatible AI metrics export from the Captain observability API for Grafana/alerting integration
- product-native AI alert evaluation derived from release-gate checks, exposed through observability JSON and Prometheus metrics
- normalized tool result envelope for audit, safety, and event telemetry without changing tool output behavior
- native trace payload policy with account-level `trace_input_capture` and `trace_output_capture` toggles, sanitizer-backed content export, and redaction hooks for OTel/Langfuse traces
- unified tool failure contract across built-in tools, copilot services, runtime wrapper, custom HTTP execution, audit logs, and event telemetry so operational failures are no longer masked as successful string outputs
- a first dataset-driven eval/regression layer for the highest-risk internal classifier flow (`captain.conversation_completion`), with reusable fixture loading, suite reporting, and a native rake entrypoint for gated runs
- deterministic offline eval packs for moderation degradation modes and tool safety blocking, runnable through native rake tasks without live provider calls
- a native combined release-check runner that merges operational `llm_events` release gating with deterministic eval packs and live high-risk eval suites into one machine-readable pre-release report
- wrapper-level observability for the main RubyLLM surfaces:
  - `Llm::ChatClient`
  - `Llm::ChatRequestRunner`
  - `Llm::ApiClient.embed`
  - `Llm::ApiClient.transcribe`
  - `Llm::ApiClient.moderate`
- normalized observability payloads through `Llm::ObservabilityPayload`, including provider/model resolution, token extraction, vector shape, transcription size, moderation outcome, and error enrichment
- persisted event ingestion for chat, embedding, transcription, moderation, schema, safety, run, handoff, and tool events
- Prometheus metrics for chat requests plus non-chat AI surfaces:
  - embeddings
  - transcriptions
  - moderation checks
  - all persisted AI tokens
  - all persisted AI estimated cost
- moderation enabled by default for assistant and copilot runtime settings, with explicit `fail_open` degradation as the default safety mode
- production-safe MCP stdio policy:
  - stdio transport is disabled by default in production
  - `CAPTAIN_MCP_STDIO_ENABLED` can explicitly enable it
  - `CAPTAIN_MCP_STDIO_COMMAND_ALLOWLIST` can restrict allowed stdio commands
- selective MCP implementation for product-managed external tools, including account-scoped server records, HTTP transports, OAuth token storage, server-config redaction, and tool metadata projection
- stricter `ruby_llm-schema` normalization so schema definitions are JSON-normalized before validation and symbol/string key mismatches do not reject valid model output

The remaining sections below still describe the target end-state and the next implementation slices.

## Current Onelink Foundation To Keep

The strongest current pieces should be kept and strengthened:

- `lib/llm/config.rb`
  - provider isolation through `RubyLLM.context`
  - account and installation model selection
- `lib/llm/chat_client.rb`
  - common chat builder and content-aware ask path
- `lib/llm/chat_request_runner.rb`
  - common request path for non-agent features
- `enterprise/app/services/llm/base_ai_service.rb`
  - thin service base aligned with the new RubyLLM path
- `enterprise/lib/captain/runtime/*`
  - product-native orchestration on top of RubyLLM, not parallel to it
- `enterprise/lib/captain/tool_registry.rb`
  - shared built-in tool metadata source
- `lib/integrations/llm_instrumentation*`
  - initial OTel/Langfuse-compatible observability foundation
- `lib/opentelemetry_config.rb`
  - production trace export foundation

The weakest current pieces are:

- split execution paths between newer `lib/llm/*` and older integration helpers
- soft schema handling instead of strict structured output contracts
- incomplete multimodal/message history fidelity
- moderation that is effectively OpenAI-only and fail-open
- tool permission behavior that is too permissive in `agent` scope
- observability that is useful but not yet normalized enough for operations and evals

## Target Architecture

## 1. One AI Platform Layer

All AI execution should converge behind one internal platform layer:

- `lib/llm/config.rb`
- `lib/llm/models.rb`
- `lib/llm/api_client.rb`
- `lib/llm/chat_client.rb`
- `lib/llm/chat_request_runner.rb`
- new:
  - `lib/llm/capability_policy.rb`
  - `lib/llm/message_serializer.rb`
  - `lib/llm/history_restorer.rb`
  - `lib/llm/schema_contract.rb`
  - `lib/llm/event_bus.rb`
  - `lib/llm/safety_policy.rb`

Product services should consume this layer.

They should not each invent:

- their own content normalization
- their own schema fallback logic
- their own trace payload shape
- their own moderation decision policy

## 2. One Product Orchestration Layer

Captain runtime should remain the orchestration layer for:

- assistant root agent
- scenario handoffs
- tool registration and execution context
- persisted conversation state
- product-specific tracing callbacks

This means:

- keep `enterprise/lib/captain/runtime/*`
- do not replace it with direct `RubyLLM::Agent`
- but reduce duplicated logic outside this runtime

## 3. One Structured Output Layer

Every feature that expects structured output should use:

- a named `RubyLLM::Schema`
- one parse-and-validate contract
- one invalid-response policy

No feature should:

- silently accept malformed shape because it is "Hash-like"
- degrade into string parsing unless the feature explicitly defines that fallback
- ship `reasoning` in user-facing product payloads unless that payload is internal-only

## 4. One Safety Layer

Safety should become a first-class platform concern, not an afterthought around the main request:

- base moderation
- custom safety rules
- redaction
- tool mutation rules
- trace capture privacy rules

Safety should apply consistently to:

- user input
- AI output
- tool arguments
- tool results before persistence when needed
- exported trace content when capture is enabled

## Detailed Workstreams

## A. Core Runtime Consolidation

### Problem

There are still multiple AI execution styles in the codebase:

- the newer RubyLLM-based service path in `enterprise/app/services/llm/base_ai_service.rb`
- the common runner in `lib/llm/chat_request_runner.rb`
- the Captain runtime path
- older integration-oriented wrappers in `lib/integrations/llm_base_service.rb`

### Decision

Converge on one platform contract:

- `Llm::ChatClient` for constructing chat
- `Llm::ChatRequestRunner` for normalized non-agent runs
- `Captain::Runtime` for orchestration-heavy runs

### Changes

- refactor `lib/integrations/llm_base_service.rb` to delegate to the same platform contract
- stop embedding model/provider/runtime semantics inside feature services
- enforce one canonical `message` shape across assistant, copilot, tasks, and helper paths

### Definition Of Done

- no major AI feature uses a bespoke request assembly path
- no major AI feature bypasses common tracing or schema handling

## B. Capability Model And Routing

### Problem

`onelink` currently models almost no capabilities besides `reasoning`.

### Decision

Promote model capabilities to first-class runtime input.

### Capability Set To Support

- `structured_output`
- `tool_calling`
- `streaming`
- `multimodal_input`
- `reasoning`
- `embedding`
- `transcription`
- `moderation`

### Changes

- expand `config/llm.yml`
- extend `lib/llm/models.rb`
- add `Llm::CapabilityPolicy`
- gate calls such as:
  - `.with_schema`
  - tool attachment
  - multimodal content
  - thinking options

### Why

This prevents:

- attaching schema to unsupported models
- attaching tools to models that should not be used that way
- silently sending multimodal content to a text-only path

## C. Message Model And Multimodal Fidelity

### Problem

Captain runtime history restoration is stronger than the generic path, and both still reduce some multimodal content to coarse text or JSON strings.

### Decision

Create one canonical message serializer/restorer shared by all AI paths.

### Changes

- extract a shared serializer from:
  - `enterprise/lib/captain/runtime/history_restorer.rb`
  - `enterprise/lib/captain/runtime/message_extractor.rb`
  - `lib/llm/chat_request_runner.rb`
- preserve:
  - role
  - content parts
  - attachments
  - tool calls
  - tool results
  - thinking
  - usage
  - agent_name when relevant

### Definition Of Done

- same input history restores equivalently in assistant and non-assistant flows
- multimodal content does not degrade differently by execution path

## D. Structured Output Hardening

### Problem

Schema usage is currently soft. Many services simply accept any `Hash`, fall back quietly, or return empty arrays on invalid structured output.

### Decision

Introduce one schema contract layer:

- build schema
- attach schema
- validate result shape
- retry or repair when invalid
- persist normalized output only after validation

### Changes

- add `Llm::SchemaContract`
- use it from:
  - Captain assistant response handling
  - Copilot response handling
  - conversation completion
  - FAQ generation
  - onboarding extraction
  - contact extraction helpers
- expand current schema usage to include:
  - `integer`
  - `optional`
  - `one_of`
  - `define/reference`
  - schema composition for repeated sub-objects

### Definition Of Done

- schema-invalid responses are visible, measurable, and retryable
- product features no longer silently degrade because the model returned the wrong shape

## E. Tooling Core

### Goal

Make built-in tools, custom HTTP tools, and future external tools behave as one logical system.

### Keep

- `enterprise/lib/captain/tool_registry.rb`
- `enterprise/app/models/captain/custom_tool.rb`
- idempotency and audit direction already present

### Needed Changes

- define one normalized tool result shape:
  - `success`
  - `data`
  - `message`
  - `error`
  - `retryable`
  - `audit`
- require mutation tools to declare:
  - risk level
  - idempotency behavior
  - confirmation behavior
  - audit fields
- tighten `Captain::ToolPolicy`
  - do not automatically bypass permission checks in `agent` scope for risky write operations
- add trace fields:
  - tool class
  - tool id
  - built-in vs custom
  - execution latency
  - result size
  - retryable failure or hard failure

### Non-Goal

Do not redesign tool UX or product concepts before this runtime hardening is complete.

## F. MCP Boundary

### Recommendation

Keep MCP as a controlled external tool-provider boundary, not as the core AI runtime.

### Current Position

- `Captain::McpServer` provides account-scoped MCP server configuration
- HTTP transports (`streamable`, `sse`) are the preferred SaaS path
- OAuth metadata and tokens are stored through the product model instead of a generic engine store
- API output redacts secrets from server configuration
- MCP tool metadata is projected into the Captain tool model instead of creating a parallel tool system
- stdio is disabled by default in production and must be explicitly enabled with `CAPTAIN_MCP_STDIO_ENABLED`
- stdio commands can be restricted with `CAPTAIN_MCP_STDIO_COMMAND_ALLOWLIST`

### What To Avoid

- do not make MCP tools bypass `Captain::ToolPolicy`
- do not expose MCP prompts/resources until product requirements justify it
- do not let stdio become a tenant-configurable production escape hatch
- do not create a second observability or audit path for MCP

### When To Revisit

Only revisit when one of these is true:

- enterprise customers need broader MCP resource or prompt support
- an external ecosystem around MCP tools becomes a product advantage for Captain
- custom HTTP tools become insufficient for important integrations

### Constraint

If MCP arrives later, it must pass through:

- `Captain::ToolPolicy`
- `Llm::EventBus`
- audit logging
- moderation/safety checks where relevant
- tracing and cost segmentation

## Custom Moderation

Base provider moderation is not enough for the actual product.

Add product-native safety rules for:

- PII and secret leakage in traces

- disallowed vertical-specific content
- account-specific blocked topics or instructions
- custom blocklists and allowlists

## J. Evals

### Recommendation

Add evals, but only after runtime contracts are normalized.

The first shipped regression suite now exists for `captain.conversation_completion`:

- fixture path: `config/llm_evals/conversation_completion.yml`
- native runner: `bundle exec rake llm:evals:conversation_completion ACCOUNT_ID=...`
- intended use: prompt/model regression checks for the internal auto-resolution classifier before operational rollout changes

The deterministic offline safety pack now also exists:

- moderation fixtures: `config/llm_evals/moderation.yml`
- tool safety fixtures: `config/llm_evals/tool_safety.yml`
- offline runner: `bundle exec rake llm:evals:deterministic`
- full pack runner: `bundle exec rake llm:evals:all ACCOUNT_ID=...`
- intended use: pre-release regression checks for moderation degradation modes, custom blocklists, and tool safety blocking without depending on live provider calls

The combined release check now exists as a separate native operator path:

- runner: `bundle exec rake llm:release_check ACCOUNT_ID=...`
- admin API: `GET /api/v1/accounts/:account_id/captain/observability/release_check`
- optional filters: `FEATURE=assistant MODEL=gpt-5.2 RUNTIME_MODE=captain_runtime ASSISTANT_ID=... SINCE=... UNTIL=... EVAL_MODEL=...`
- API note: the HTTP endpoint defaults to `include_live=false` and enables account-scoped live evals only when explicitly requested
- behavior: fails only on blocking operational gate or eval failures; `pass_with_warnings` remains a successful run with advisory warnings

### What To Evaluate

- assistant response correctness
- tool selection correctness
- schema validity

- handoff behavior
- conversation completion classification

### What To Store

- moderation is explicit, measurable, and not silently skipped
- traces are complete enough to debug a run end-to-end
- metrics and alerts exist for latency, errors, cost, moderation, and schema health
- eval datasets exist for the highest-risk flows
- new AI features naturally plug into the same runtime and safety envelope instead of inventing their own path

## Non-Goals

- matching every capability in all neighboring RubyLLM repositories
- adding MCP because it is fashionable
- adding new UI surfaces before the runtime contracts are stable
- treating internal reasoning as a user-facing product field

The right end state is not maximal breadth.

The right end state is a narrow, coherent, RubyLLM-native AI platform that fully covers `onelink`'s real product use cases and is operationally safe to evolve.
