# Internal Data Model And Database

## Database Shape

The application uses a relational schema with first-class tables for stable business entities and JSONB only for controlled variability.

`db/schema.rb` is the current schema reference and `db/migrate/` shows the evolution of major subsystems such as Captain, scheduling, CRM, and assignment policies.

## Main Design Rules

1. Stable business lifecycle belongs in real tables and columns.
2. Tenant-specific or flexible fields belong in controlled JSONB attribute containers.
3. `account_id` is the default tenant boundary across runtime entities.
4. Indexes are added for active list dimensions, uniqueness, and JSONB search where needed.

## JSONB Conventions

The codebase uses JSONB intentionally in several places:

| Field pattern | Typical purpose |
| --- | --- |
| `custom_attributes` | Flexible business metadata attached to an entity |
| `additional_attributes` | System or integration metadata |
| `settings` | Account or subsystem settings |
| `internal_attributes` | Internal account-level metadata |
| `config` / `metadata` / `rules` / `options` | Structured subsystem configuration |

## Flexible Field Systems

There are two field-definition models in the current runtime:

### Legacy Custom Attributes

- `CustomAttributeDefinition`
- used for `Contact` and `Conversation`

### Managed Runtime Field Definitions

- `Crm::FieldDefinition`
- used for `deal`, `task`, and `appointment`

This split matters when extending the data model or designing filters, automations, and UI rendering.

## Internal Rules For Schema Changes

1. Prefer a first-class table when the lifecycle is stable and relational.
2. Prefer JSONB only when the shape is tenant-specific or extensible.
3. Add indexes for active list surfaces and high-volume filters early.
4. Keep account-scoped uniqueness explicit in DB constraints where practical.
